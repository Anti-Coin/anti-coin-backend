# Coin Predict Project Identity

- Last Updated: 2026-02-12
- Status: Active

## 1. 왜 이 프로젝트를 하는가
1. 단순 데모가 아닌 운영 가능한 AIOps 역량을 축적한다.
2. 실제 장애/복구/관측 경험을 통해 엔지니어링 실력을 체계화한다.
3. 장기적으로 확장 가능한 서비스 기반을 만든다.

## 2. 오너 철학 (비교적 안정적)
1. 안정성을 기능보다 우선한다.
2. 한 번에 크게 바꾸지 않고 작은 단위로 전진한다.
3. 계획은 고정이 아니라 Living Plan으로 관리한다.
4. 비판적 토론 후 구현한다.
5. 운영 가능한 단순한 해법을 우선한다.
6. 문서는 기록이 아니라 협업 인터페이스다.

## 3. 도메인 정책 (철학과 분리, 변경 가능)
`비트코인 가격` 도메인 특성 때문에 현재 정책으로 채택한 항목들:
1. stale 데이터는 상태를 숨기지 않고 명시한다.
2. soft stale은 경고와 함께 노출할 수 있다.
3. hard stale/corrupt는 차단한다.
4. 내부 시간 처리는 UTC 기준으로 통일한다.

주의:
1. 위 항목은 "보편 철학"이 아니라 "도메인 정책"이다.
2. 도메인이 바뀌면 정책도 변경될 수 있다.

## 4. 비전/목적/지향성과 함께 관리할 가치
1. Risk Appetite: 어디까지 실패를 허용하는지
2. Evidence Standard: 무엇을 완료 증거로 볼지
3. Change Cadence: 변경 속도와 배치 기준
4. Cost Ceiling: 리소스/운영 비용 상한
5. Recovery Philosophy: fail-open/fail-close 기준
6. Ownership: 누가 최종 판단하고 책임지는지

## 5. 비목표 (현재 시점)
1. 단일 스프린트 내 전 기능 동시 최적화
2. 초기 단계에서 무거운 MLOps 플랫폼 도입
3. 1분봉 예측 서빙을 당장 제공

## 6. AI 협업 정렬 규칙
1. 새 세션 시작 시 이 문서를 먼저 읽고 이해를 5줄 요약한다.
2. 작업 제안 시 철학 항목과 도메인 정책을 분리해서 근거를 제시한다.
3. 구현 후에는 변경 내용과 잔여 리스크를 문서에 남긴다.
4. 코드 가독성을 위해 복잡한 로직에는 의도 주석을 적극적으로 남긴다.
5. 운영 추적이 필요한 상태전이는 로그로 명시하고, 침묵 실패를 만들지 않는다.
